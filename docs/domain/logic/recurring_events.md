# Recurring events

A recurring event must always have an explicit IANA timezone in `events.recurrence_timezone`.

Examples of valid values:

```text
Europe/Paris
America/New_York
Asia/Tokyo
```

Fixed offsets such as `UTC+1` must not be used as recurrence timezones because they do not describe daylight-saving transitions.

The default timezone proposed when creating a recurring event may be resolved from organization-level settings, but the resolved timezone must be copied into the recurring event itself.

Changing an organization or schedule setting must not implicitly change the timezone of an existing recurring event.

`events.recurrence_rule` contains only the recurrence pattern and its boundaries. The timezone and the initial local start and end are stored in dedicated columns.

## Changes to recurrence rules

Any change that alters the theoretical occurrence set must trigger an impact analysis before persistence.

Examples:

* changing weekdays;
* changing frequency;
* changing interval;
* changing the recurrence start;
* changing the recurrence boundary;
* changing the recurrence timezone.

The impact analysis must identify exceptions whose original occurrence is no longer generated by the new rule.

No incompatible exception may be silently migrated or deleted.

Initial resolution choices should be:

* cancel the recurrence-rule update;
* delete incompatible exceptions;
* convert modified incompatible exceptions into unique events.

A cancelled exception does not produce a unique event during conversion.

A modified exception converted into a unique event must materialize its final visible state, including inherited values and overrides.

The recurrence update, exception resolution and unique-event creation must occur in a single transaction.

Events and recurring-event occurrences become immutable once their effective end instant has passed.
Changes to recurring series apply only to future occurrences.
